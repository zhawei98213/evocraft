# EvoCraft 种子想法胶囊

创建日期：2026-05-10

## 胶囊记录铁律

任何新的 PRD、PRD 重要更新、明确产品想法，都必须同步进入这个想法胶囊。

记录方式：

- 按日期写入时间线。
- 提炼想法、原则、观点和产品判断。
- 不复制 PRD 原文。
- PRD 负责结构化需求，想法胶囊负责记录产品意图和演化脉络。

## 想法时间线

### 2026-05-18

- EvoCraft 的总品牌 logo 应优先表达“应用集合 + AI 整理 + 成长 Craft”，而不是被单一的拍照、扫描或错题功能锁死；错题收集可以拥有更直接的扫描笔记子应用图标。
- 品牌视觉可以沿用当前工作台基线的可信蓝和青绿色，但需要用少量暖黄色保留孩子成长感；游戏化只能作为轻点缀，不能让产品显得像玩具。
- 错题收集子应用图标应优先走“扫描框 + 打开笔记 + 整理完成”方向：它直接承接拍照错题到干净题面的核心价值，同时需要减少真实文字和复杂页面细节以保证小尺寸可信。
- 当前 logo 选定稿确认扫描笔记方向：品牌第一眼应让家长和孩子理解“AI 把拍下来的错题整理成可复习资料”，而不是先强调抽象平台、游戏或相机能力。
- Logo 从设计稿进入产品壳时，应同时覆盖应用内品牌位、App Hub 功能入口、renderer favicon 和桌面 app icon；视觉资产不能只停留在设计文档里。

### 2026-05-17

- 桌面壳路线改为 Electron-first：第一版桌面产品优先追求 Chromium 渲染一致性、JS/TS 工程连续性和桌面调试效率，而不是优先压缩安装体积。
- Electron 的能力必须和安全边界一起设计：renderer 不直接获得 Node 权限，桌面文件访问通过 preload + IPC 白名单进入 main process，儿童学习照片相关能力默认最小暴露。
- 设计源文件要以当前可运行产品为准：已废弃的 Figma SVG 导入包不再恢复，后续视觉回归和桌面迁移对齐以 implemented-mvp 与 desktop-trunk 截图基线为准。

### 2026-05-16

- 原型收尾的重点不是新增学习能力，而是把真实试用会破坏信任的边界补齐：隐私确认、可删除本地数据、失败后能继续。
- 错题照片属于儿童学习敏感材料；即使当前只是本地 mock，也要在进入处理前让用户明确知道照片没有上传，未来外部 AI 必须单独授权。
- 技术路线应跟随复杂度，而不是提前换栈：错题收集 MVP 先继续静态 Web，等 AI adapter、复习/分析页面和共享组件增多后再迁移 React/Vite；桌面壳和后端等明确持久化或同步需求后再启动。
- PRD 规范本身也要从实际 PRD 中学习：当既有 PRD 有规范尚未提炼的有效结构或产品原则时，不能为了套模板而删弱内容，而要把它反向沉淀进 PRD 编写规范。
- 后续 PRD 应成为跨产品、设计、工程和测试的统一交接物：不仅写功能，还要写默认决策、非功能需求、AI/隐私/数据边界、风险、验收和设计实现交接。
- 技术路线决策前必须先把真实实现固化成 UI 设计图：决策应面对当前页面表面积、交互复杂度和视觉回归风险，而不是只看 PRD 或口头印象。
- 静态 Web 可以继续作为可试用 MVP，但下一步价值不在继续堆页面，而在把 AI/OCR 能力抽成 provider-agnostic adapter，并用 mock contract tests 保护真实模型接入前的业务边界。
- MVP 完成后，技术路线的重心从“是否换栈”转为“桌面版如何交付”：静态 Web 保留为基线，真正的下一阶段主干应是 React/Vite/TypeScript + AI adapter，再接 Tauri 桌面壳。
- 多平台不是同一套界面硬适配：桌面版负责精细整理和本地资料库，平板更适合触控复核和练习，手机更适合快速拍照采集和轻量复习。

### 2026-05-15

- `选择题目区域` 不只是新增框选能力，还需要可撤销的候选框管理：AI 框错时用户应能删除干扰框，避免后续手动画框和候选框叠在一起造成误选。
- 删除候选框后，流程必须保持可恢复：删光候选框不等于流程失败，而是引导用户手动画框或重新自动找题。

### 2026-05-13

- 真实拍照经常包含多道题，错题收集的第一版不能假设一张照片天然等于一道题；必须先让用户确认本次识别的题目区域，再进入识别和去痕。
- `选择题目区域` 应成为独立且顺滑的核心步骤：AI 先给候选框，用户可以微调，也可以在 AI 没框中时手动画框；第一版不做整卷批量拆题。
- 保存错题时应同时保留整张原图和确认后的题目区域；错题本默认使用干净题面，但详情必须能回到原始照片语境。
- 真实 AI 接入不应默认追求最贵旗舰模型；EvoCraft 第一条生产链路应国内模型优先、Qwen 体系优先评估、按题目难度分层调用，并保持 provider-agnostic，避免成本和访问稳定性压垮 MVP。

### 2026-05-11

- 错题收集闭环不能只停在“保存成功”和单条详情，必须有明确的 `错题本` 入口承载全部已保存记录；否则用户完成收集后不知道资料沉淀在哪里。
- `错题本` 是错题收集和后续复习、分析之间的桥梁：列表默认呈现干净题面状态，详情再提供原图溯源，避免把复习入口和上传入口混在一起。
- 当用户明确指出缺失功能或要求补功能时，这不是单纯实现任务，而是产品需求边界发生变化；必须同步更新 PRD 和想法胶囊，让代码、需求和产品意图保持一致。

### 2026-05-10

- EvoCraft 的长期形态是 AI 学习助手应用集合，不是单一错题工具。
- 第一版 MVP 只让 `错题收集` 成为可用应用，但顶层需要保留 App Hub 和后续应用扩展空间。
- 错题收集的核心价值不是保存照片，而是把带有书写和批改痕迹的真实错题整理成可复习的干净题面。
- 原图和干净题面必须并存：原图用于溯源和复核，干净题面用于复习、分析和后续练习。
- 第一版采用本地优先和 AI provider-agnostic 思路；UI 阶段可以先 mock 识别结果，不应过早绑定具体模型服务。
- 儿童学习照片属于敏感材料，真实外部 AI/OCR 上传必须在隐私、授权、存储、删除边界明确后再做。
- 后续每次 PRD 或明确想法变化，都要把观点提炼回这个胶囊，并按时间线保留演化记录。
- 长期开发不能依赖聊天上下文；项目记忆、进度和想法演化必须写入仓库文档，并在进度记录里保留任务、完成情况、卡点、执行命令和下一步。
- 第一版 UI 原型应以桌面端工具感为主，使用 App Hub 展示应用集合身份，用四屏流程呈现上传、识别检查、去痕复核和保存结果。
- UI 风格选择先做横向探索：清爽学习桌、未来学习工作台、纸面笔记本、成长收集册四类方向都要服务同一个错题收集流程，不能为了风格牺牲可复核和可信任。
- 已选择 B 方案 `未来学习工作台` 作为第一版视觉基线：EvoCraft 应先像一个高效、可信的学习工作台，而不是普通拍题软件或游戏大厅。蓝色主色用于表达 AI 整理、结构化和可信复核，青绿色只作为辅助提示，清晰状态标签用于降低家长和孩子复核 AI 结果的成本。
- 正式 UI 图要为 coding 服务：优先表现可拆分的应用外壳、上传区、原图/干净题面、AI 复核面板、状态 chip 和保存详情，而不是只做风格氛围图。
- 进入 Figma 前需要把 UI 图转成可编辑的矢量参考稿：总览图用于确认流程，独立页面 SVG 用于拆组件、建 Auto Layout 和后续编码对齐。

## 原始想法

EvoCraft 是一个 AI 学习助手应用集合，第一批面向上海的小朋友。它不应该一开始就做成庞大的完整平台，而是先保留应用集合的顶层结构，再做出第一个最小但完整的应用闭环。

第一版 MVP 是 EvoCraft 应用集合里的第一个应用：`错题收集`。用户上传一张包含一道错题的照片，应用识别题目内容，支持语文、数学、英语，并且在图形题、公式、图表、几何图或识别不确定时保留视觉内容。因为真实错题通常带有学生书写、批改、圈画、涂改等痕迹，应用需要生成适合复习的干净题面，而不是直接把带作答痕迹的照片当成最终材料。

后续 EvoCraft 可以逐步扩展成更完整的学习助手应用集合：错题分析、复习、背单词、阅读辅助、游戏元素、金币收集、收藏循环、虚拟人物装扮等。

## 应用集合结构

EvoCraft 顶层需要保留 `App Hub / 应用集合首页` 的思路。第一版只让 `错题收集` 可用，但产品和 UI 不应把 EvoCraft 设计成只能做错题的单一工具。

未来应用可以包括：

- 背单词。
- 复习计划。
- 阅读辅助。
- 数学专项练习。
- 虚拟人物和装扮。
- 学习奖励与金币。

## 产品气质

EvoCraft 应该是“认真学习工具 + 轻度游戏化成长感”的结合，而不是普通拍题软件。名字里的 `Craft` 暗示孩子在逐步打造自己的学习资料、学习世界和成长路径。

## 第一个能力碎片

错题收集应用：

- 上传或拖入一张照片。
- 选择或自动判断科目。
- 从图片中识别一道题。
- 尽可能提取文字。
- 对图形、公式、几何图、表格等内容保留图片片段。
- 去除或隔离学生书写、批改、圈画等痕迹，生成干净题面。
- 用户可以修正识别结果。
- 保存一个干净的错题记录，默认展示去痕后的题面，原图保留用于溯源和重新处理。

## 长期能力碎片

- 按知识点和错误原因分析错题。
- 复习计划和练习循环。
- 生成相似题。
- 给家长或老师看的简要总结。
- 背单词应用。
- 金币和收藏物。
- 虚拟人物装扮。
- 孩子的学习进度地图。
- 桌面、平板、手机多端体验。

## 设计种子

- 桌面端优先，因为第一版需要仔细检查照片识别结果。
- 第一屏可以是轻量应用集合入口，明确当前可用应用是“错题收集”。
- UI 要让“不确定”可见：识别文本和图片片段可以并存。
- UI 要区分“原图”和“干净题面”，避免孩子复习时看到自己的错误作答痕迹。
- 不要让孩子面对太多复杂控制。
- 家长可以协助上传和修正，但保存后的错题记录应该让孩子也能看懂。
- 第一版最重要的是让一个错题记录显得完整、可信、可继续使用。

## 可升级为需求的条目

后续可以分别升级为 PRD 或设计文档：

- MVP PRD：EvoCraft 应用集合中的单道错题收集应用流程。
- UI 简报：App Hub、错题收集应用入口、桌面端屏幕结构和视觉方向。
- 识别规格：OCR、科目判断、图形保留、书写痕迹去除、置信度处理。
- 数据模型规格：错题记录字段和存储方式。
- 游戏化规格：金币、奖励、虚拟人物、复习循环。
