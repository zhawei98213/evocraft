# EvoCraft PRD 编写规范

最后更新：2026-05-16

文档状态：v1.1，作为 EvoCraft 后续 PRD 新建、重大更新和需求评审的统一写作规范。

版本号：v1.1

当前维护者：EvoCraft product agents

相关文档：

- 项目级执行规则：`AGENTS.md`
- 项目记忆：`docs/planning/evocraft-project-memory.md`
- 路线图与进度：`docs/planning/evocraft-roadmap-progress.md`
- 想法胶囊：`docs/ideas/2026-05-10-evocraft-seed-capsule.md`
- 当前 MVP PRD：`docs/prd/2026-05-10-wrong-question-capture-mvp-prd.md`

适用范围：

- 新产品或新应用的 PRD。
- 现有 PRD 的重大范围更新。
- 用户明确指出缺失功能、要求补功能、指出流程缺口时的 PRD 同步更新。
- AI 能力、数据模型、隐私策略、关键页面和核心流程的需求说明。
- 发现既有 PRD 中存在本规范未覆盖、但对后续 PRD 有复用价值的结构或原则时，对本规范的反向提炼更新。

不适用范围：

- 只改错别字、格式、链接的轻量文档修补。
- 只记录灵感、观点、产品原则的想法胶囊条目。
- 纯技术实现计划。实现计划应放在 `docs/superpowers/plans/` 或等价计划文档中。

## 1. 调研结论

主流 PRD 规范没有唯一模板，但共同目标高度一致：让产品、设计、工程、测试和利益相关者对“为什么做、为谁做、做什么、不做什么、怎么验收”达成同一理解。

本规范参考了以下公开资料，并结合 EvoCraft 已有文档机制提炼：

- Atlassian / Confluence 的 PRD 指南和模板：强调目标、成功指标、假设、用户故事、设计、范围和 out-of-scope。
- Aha! 的需求管理与 PRD 模板：强调需求管理是持续过程，需要收集、分析、定义、优先级和验证，并区分 MRD 与 PRD。
- Productboard 的 PRD 说明：强调 outcome、success measures、stakeholders、personas、scope、release、assumptions、risks、constraints、dependencies。
- Pendo 的 PRD 模板：强调 living guide、目标、用户、关键功能、技术和设计约束、里程碑、测试、发布和迭代计划。
- Miro 的 PRD 模板：强调 PRD 是跨团队共享理解工具，应覆盖 overview、goals、success metrics、target audience、features、UX、assumptions、dependencies、out of scope、owner、version。
- NASA 需求编写指南和软件需求手册：强调需求必须清晰、无歧义、完整、一致、可验证、可追踪，并且每条需求应尽量只表达一个要求。

提炼后的 EvoCraft PRD 原则：

1. PRD 先解释问题和价值，再定义功能。
2. PRD 写“产品需要达成什么”，不提前锁死不必要的技术实现。
3. 每个功能需求都要能追溯到用户目标、场景或产品原则。
4. 每个关键需求都要有可验证的验收标准。
5. 范围和非目标同样重要，非目标用于保护 MVP。
6. 风险、假设、依赖和开放问题必须显性化。
7. PRD 是活文档；需求边界变化时要同步更新 PRD、想法胶囊和进度记录。
8. 既有 PRD 也是规范来源；如果 PRD 中有本规范尚未提炼的有效结构、产品边界或验收原则，必须在同一变更中更新本规范。
9. EvoCraft PRD 必须保留应用集合视角，避免把阶段性功能误写成整个产品终点。
10. 涉及 AI/OCR、外部服务或儿童学习照片时，PRD 必须写明供应商抽象、模型分层、官方文档复核、隐私授权和删除边界。
11. PRD 必须给设计、工程和测试留下交接输入，包括 UI 生成摘要、实现切片、测试重点和需要同步更新的文档。

## 2. PRD 分层

不同任务需要不同深度，不能所有需求都写成同样重量的文档。

| 类型 | 适用场景 | 最低要求 | 推荐位置 |
| --- | --- | --- | --- |
| 概念 PRD | 新应用、新方向、新阶段前的收敛 | 问题、目标用户、价值、范围、非目标、成功标准、开放问题 | `docs/prd/` |
| MVP PRD | 第一个可用闭环或阶段性版本 | 完整标准结构，必须包含流程、页面、数据、验收、风险 | `docs/prd/` |
| 功能 PRD | 已有产品中的核心功能新增或大改 | 背景、目标、用户故事、流程、需求、验收、影响范围 | `docs/prd/` |
| AI 能力 PRD | OCR、多模态、模型链路、自动分析等 | 输入输出、置信度、失败状态、人工复核、隐私、成本边界、可替换供应商 | `docs/prd/` 或 `docs/planning/` |
| PRD 增量更新 | 用户指出缺失功能、范围变更、关键策略变化 | 更新原 PRD 相关章节，保留版本和变更说明，同步想法胶囊和进度 | 原 PRD 文件 |

检查点：

- 当前需求是否真的需要新 PRD，还是更新已有 PRD 更合理？
- 如果是已有能力的边界变化，是否已经定位到应更新的原 PRD？
- 如果是产品方向或明确产品想法变化，是否需要同步想法胶囊？

## 3. 标准 PRD 结构

### 3.1 元信息

必须包含：

- 文档标题。
- 最后更新日期。
- 文档状态，例如草稿、评审中、已确认、已废弃。
- 版本号。
- 文档责任人或当前维护者。
- 变更摘要，说明本次版本为什么更新。
- 相关文档链接，例如想法胶囊、设计稿、实现计划、代码入口。

检查点：

- 标题是否能让人一眼知道产品、能力和阶段？
- 版本号是否和正文变更一致？
- 日期是否是实际更新日期？
- 变更摘要是否说明了范围变化，而不是只写“更新”？
- 是否能从这里跳到相关上下文？

### 3.2 产品定位

回答这个 PRD 服务的产品是什么，以及它在 EvoCraft 中的位置。

必须写清：

- 这是 EvoCraft 整体、某个学习应用，还是某个应用内功能。
- 该能力和当前路线图阶段的关系。
- 它对长期产品形态的影响。
- 如果是某个应用内能力，它和 EvoCraft 应用集合、App Hub 或未来学习应用的关系。

写作标准：

- 用 1 到 3 段说明，不要写营销文案。
- 如果只是阶段性功能，要明确它不是产品终点。
- 如果会影响信息架构、AI 策略、隐私策略或数据形态，要提前标注。

检查点：

- 读者是否知道这个需求属于哪个应用、哪个阶段？
- 是否避免把局部功能误写成整个产品定位？
- 是否避免把 EvoCraft 收窄成单一错题工具或单一页面功能？
- 是否和 `docs/planning/evocraft-project-memory.md` 一致？

### 3.3 背景与问题

说明用户遇到什么问题，以及为什么现在值得做。

必须包含：

- 真实用户场景。
- 当前痛点或机会。
- 不解决会发生什么。
- 已有证据来源，例如用户反馈、项目记忆、想法胶囊、测试观察、业务判断。

写作标准：

- 先写问题，不先写解决方案。
- 避免“用户需要一个按钮”这类把方案伪装成问题的表述。
- 问题必须能对应后续目标和验收标准。

检查点：

- 问题是否来自用户、场景或明确产品原则？
- 是否能说清这是刚需、体验改进、效率提升，还是战略铺垫？
- 是否把假设和事实分开？

### 3.4 目标

目标定义本轮要达成的结果，不是功能清单。

必须包含：

- 用户目标。
- 产品目标。
- 如果适用，业务或学习效果目标。
- 至少 1 个可验证结果。

写作标准：

- 使用结果导向句式，例如“用户可以完成上传、确认区域、识别复核、保存、回看闭环”。
- 目标数量控制在 3 到 7 条。
- 不把实现方式写成目标，除非实现方式本身是约束。

检查点：

- 每个目标是否能被后续功能或验收标准支撑？
- 是否存在不可验证的空泛目标，例如“体验更好”“更智能”？
- 是否和非目标一起保护范围？

### 3.5 非目标

非目标用于明确本轮不做什么。

必须包含：

- 不进入本版本的功能。
- 不承诺的自动化程度。
- 不改变的系统边界。
- 不处理的用户群体、平台或场景。

写作标准：

- 非目标要具体，不能写成“其他不相关功能”。
- 如果某件事未来可能做，写“本版本不做”，不要写“永远不做”。
- 非目标应能防止实现阶段范围膨胀。

检查点：

- 是否列出了最容易被误以为要做的能力？
- 是否明确哪些能力只是占位、mock 或后续阶段？
- 是否能帮助工程判断哪些请求应延后？

### 3.6 用户与场景

说明谁会用、在什么情况下用、成功体验是什么。

必须包含：

- 主要用户。
- 辅助用户。
- 使用场景。
- 用户已有能力、限制和设备环境。
- 对未成年人、家长、老师等角色的特殊考虑。

写作标准：

- 用户角色要和行为相关，不只写人口标签。
- 场景要足够具体，能指导页面和流程。
- 不要把所有潜在用户都列为主要用户。

检查点：

- 是否能说明“谁在什么情况下为了什么结果使用这个功能”？
- 是否区分了操作者、受益者和审核者？
- 是否覆盖了桌面、平板、手机等关键平台假设？

### 3.7 决策边界与默认假设

说明本 PRD 为了推进执行而做出的默认判断，以及哪些事项还不能擅自决定。

必须包含：

- 已确认决策。
- 可由实现阶段自行决定的细节。
- 需要再次确认的高风险事项。
- 关键假设及其验证方式。
- 如果技术路线会影响产品边界、隐私、成本或后续迭代，必须写明当前技术默认值和迁移触发点。

写作标准：

- 默认假设要明确可撤销。
- 涉及真实儿童数据、外部模型、账号、支付、云同步等事项，必须列入需要确认的边界。
- 技术默认值不得和产品目标冲突。
- 技术路线决策必须说明哪些是当前默认、哪些是后续触发条件，不能把临时实现写成长期承诺。

检查点：

- 哪些决定是产品决定，哪些只是实现建议？
- 是否存在需要用户或负责人再次确认的隐私、成本、供应商或平台选择？
- 假设失败时是否有降级方案？
- 是否记录了 React/Vite、Electron/Tauri、后端、账号、云同步等路线何时才应启动？

### 3.8 用户故事与任务

用户故事用于把目标转成可讨论的用户行为。

推荐格式：

```text
作为 <用户角色>，
我想要 <完成某个行为或获得某种能力>，
以便 <达成具体价值或结果>。
```

写作标准：

- 用户故事描述用户目标，不描述内部实现任务。
- 每条用户故事应能映射到一个或多个验收标准。
- 大用户故事需要拆成核心流程和子任务。

检查点：

- 这条故事的主语是真实用户，还是团队内部角色？
- “以便”是否表达了用户价值，而不是开发理由？
- 是否遗漏了失败、修改、取消、回退等关键路径？

### 3.9 核心流程

流程描述用户从开始到完成目标的路径。

必须包含：

- 主成功路径。
- 可恢复失败路径。
- 取消、返回、重试、编辑、保存、查看等状态转换。
- 如果涉及 AI，需要写清人工复核点。

写作标准：

- 用编号步骤写主流程。
- 用子弹列表补充异常和分支。
- 不把流程写成页面元素堆砌。

检查点：

- 用户从入口到完成是否闭环？
- 失败后是否能继续，而不是直接终止？
- 保存后的内容能否被找回、复查或编辑？
- AI 输出是否被当成草稿而不是绝对结果？

### 3.10 页面与信息架构

说明用户会看到哪些页面、区域或状态。

必须包含：

- 页面或主要界面列表。
- 每个页面的目的。
- 每个页面必须包含的信息和操作。
- 导航关系和返回路径。
- 空状态、加载状态、错误状态。

写作标准：

- 先写页面目的，再写元素。
- 控件、标签、列表、状态不要脱离用户任务。
- 对复杂页面要写清主次层级。

检查点：

- 每个页面是否有唯一清晰的目的？
- 用户能否知道当前位置和下一步？
- 是否存在只在最近记录里可见、没有长期入口的重要内容？
- 错误和空状态是否能引导用户继续？

### 3.11 功能需求

功能需求定义产品必须做什么。

必须包含：

- 需求编号。
- 需求描述。
- 优先级。
- 来源或对应用户故事。
- 验证方式。
- 相关页面、数据或状态。

推荐格式：

| ID | 优先级 | 需求 | 来源 | 验收方式 |
| --- | --- | --- | --- | --- |
| PRD-FR-001 | P0 | 系统必须允许用户上传一张图片并看到预览。 | 上传错题用户故事 | 浏览器手动验证或端到端测试 |

写作标准：

- 每条需求只表达一个行为或约束。
- 使用“必须 / 应该 / 可以”区分强度。
- 避免模糊词：简单、快速、智能、尽量、适当、友好。必须使用时要补充可验证标准。
- 不写无法验证的主观要求。

检查点：

- 每条需求是否必要？
- 是否单一、清晰、无歧义？
- 是否可通过测试、检查、演示或分析验证？
- 是否能追溯到目标、用户故事或产品原则？

### 3.12 非功能需求

非功能需求定义质量属性和运行边界。

常见类别：

- 性能。
- 可用性。
- 可访问性。
- 兼容性。
- 隐私与安全。
- 可靠性。
- 可维护性。
- 成本与调用频率。
- 数据保留和删除。

写作标准：

- 尽量量化，例如“上传失败时必须保留已填写字段”比“上传要稳定”更好。
- 如果当前无法量化，要写明观察方式和后续决策点。
- 不把工程偏好包装成产品要求。

检查点：

- 是否覆盖了会影响用户信任的质量要求？
- 是否覆盖了儿童学习照片、模型调用、数据留存等敏感事项？
- 是否明确了成本、供应商和降级约束？

### 3.13 数据模型

当功能会保存、传输、展示或分析数据时必须包含。

必须包含：

- 核心实体。
- 字段名、类型、含义。
- 必填和可选。
- 生命周期。
- 数据来源。
- 是否包含敏感信息。
- 和页面、AI 输出、存储方案的关系。

写作标准：

- 数据模型是产品语义模型，不等同于数据库设计。
- 字段要解释用户意义，不只写技术类型。
- 敏感字段必须标注隐私处理要求。

检查点：

- 保存后的记录是否足够支持后续查看、复习、分析和溯源？
- 是否有字段只能从聊天上下文理解，文档中没有定义？
- 原图、处理结果、AI 输出和用户编辑内容是否分开？

### 3.14 AI 能力契约

只要涉及 OCR、多模态、模型调用、生成、分类或分析，就必须写 AI 能力契约。

必须包含：

- 输入。
- 输出。
- 置信度或需复核标记。
- 用户可编辑范围。
- 失败和降级状态。
- 供应商抽象边界。
- 成本和模型分层策略。
- 隐私和数据留存要求。
- 如果写入具体供应商或模型系列，必须标明这是候选链路、备选链路或已确认链路。
- 真实接入前需要复核的官方文档、价格、权限和隐私约束。

写作标准：

- AI 输出默认是草稿，除非有独立验证机制。
- 业务层不直接绑定具体模型供应商。
- 复杂题、低置信度、失败重试要有状态。
- 不把不可编辑的模型生成结果作为唯一事实来源。
- 国内模型优先、分层调用、备选供应商和官方文档复核属于可写入 PRD 的产品/成本/可用性边界。
- 面向学生或家长的 UI 不应暴露复杂模型名、供应商术语或内部置信度细节，除非这些信息是用户做判断所必需。

检查点：

- 用户能否看出 AI 结果需要复核？
- 失败时是否仍能手动完成核心任务？
- 是否避免把高成本模型作为无条件默认链路？
- 是否能替换供应商而不重写产品流程？
- 是否说明了真实接入前要重新核对模型名、价格和能力？
- 是否避免在儿童可见 UI 中堆叠模型供应商术语？

### 3.15 隐私、安全与合规

涉及未成年人、学习照片、账号、外部服务、云同步时必须包含。

必须包含：

- 采集哪些数据。
- 为什么需要。
- 数据在哪里处理。
- 是否上传外部服务。
- 保存多久。
- 用户如何删除或撤回。
- 失败、泄露、误传的风险和缓解方式。
- 本地删除、云端删除、供应商侧删除之间的区别。

写作标准：

- 最小收集原则。
- 默认本地优先时要写清未来接入外部服务的前置条件。
- 不使用“安全合规待后续考虑”作为占位。
- 记录删除和清空本地数据是用户信任能力，不应只写在实现计划里。

检查点：

- 是否包含儿童学习照片、学校、姓名、字迹、分数等敏感信息？
- 是否明确外部 AI/OCR 调用前需要授权和提示？
- 是否能解释为什么必须处理这些数据？
- 是否说明删除行为影响的是本地、云端还是外部供应商侧数据？

### 3.16 UI 文案与语气

定义用户可见表达的边界。

必须包含：

- 主要语言。
- 面向角色。
- 语气原则。
- 禁用或慎用术语。
- AI 不确定性表达方式。

写作标准：

- 面向孩子和家长时，文案要清楚、温和、可行动。
- 不把技术词堆到用户界面里。
- 不把 AI 结果包装成绝对正确。

检查点：

- 用户能否不用懂技术就知道下一步？
- 是否避免恐吓式、营销式或过度承诺文案？
- 不确定状态是否说清楚？

### 3.17 成功指标

成功指标说明如何判断这个 PRD 是否达成目标。

可选类别：

- 行为指标：完成率、保存率、回访率。
- 质量指标：识别复核通过率、失败率、编辑率。
- 体验指标：完成时间、步骤中断点。
- 学习价值指标：复习使用次数、错题分析使用率。
- 工程指标：错误率、响应时间、模型调用成本。

写作标准：

- MVP 可以先用定性或手动验证指标，但要写清如何观察。
- 指标要服务目标，不能只列容易统计的数据。
- 如果没有埋点，也要写人工验证口径。

检查点：

- 每个核心目标是否有对应成功指标？
- 指标是用户价值导向，还是只反映团队产出？
- 是否能在当前阶段实际收集或验证？

### 3.18 验收标准

验收标准是 PRD 最重要的交付接口之一。

必须包含：

- 主流程验收。
- 关键异常验收。
- 数据保存和回看验收。
- 隐私、安全或 AI 复核验收。
- 明确不依赖哪些非目标。

推荐写法：

```text
给定用户已上传一张包含多道题的照片，
当用户确认其中一道题的区域并点击识别，
那么系统只针对确认后的区域生成识别草稿，并在保存记录中保留整张原图和确认区域。
```

也可以使用清单：

- 用户可以从 App Hub 进入错题收集。
- 用户上传图片后可以看到预览。
- 用户可以确认一道题区域。
- 保存后的记录可以从错题本找回。

检查点：

- 验收标准是否能被人手工执行或自动测试？
- 是否覆盖失败、空状态、低置信度和返回路径？
- 是否能证明核心用户价值已经达成？
- 是否避免把验收标准写成实现任务清单？

### 3.19 风险、依赖与开放问题

明确不确定性，避免实现阶段隐性踩坑。

必须包含：

- 产品风险。
- 体验风险。
- 技术风险。
- 隐私风险。
- 成本风险。
- 外部依赖。
- 未决问题。

写作标准：

- 每个风险都要有缓解策略或后续决策点。
- 每个开放问题要说明谁来回答、什么时候必须回答。
- 已经决定的事项不要继续留在开放问题里。

检查点：

- 是否把最可能阻塞实现或验收的问题列出来？
- 是否有“做着做着再说”的隐性高风险决策？
- 开放问题是否能转成后续任务？

### 3.20 实现与设计交接

PRD 不替代设计稿和实现计划，但要为它们提供清晰输入。

必须包含：

- UI 生成或设计输入摘要。
- 开发计划所需的功能切片。
- 测试重点。
- 需要同步更新的文档。
- 上线或发布前检查。
- 如果已有静态原型、Figma 包或代码入口，必须列出设计和实现对齐的引用。

写作标准：

- 写清设计应体现什么，不规定每个像素。
- 写清开发必须保留哪些产品边界。
- 写清验收证据应该来自哪里。
- UI 生成输入摘要应包含产品身份、页面主次、状态、错误恢复、隐私边界和 AI 不确定性表达。

检查点：

- 设计师或前端能否根据 PRD 拆出页面？
- 工程能否根据 PRD 拆出功能切片？
- 测试能否根据 PRD 写出验证清单？
- 后续 agent 能否知道还要同步更新哪些文档？

### 3.21 PRD 规范反向提炼与同步

当更新既有 PRD 时，如果发现该 PRD 有本规范尚未覆盖但值得复用的结构、产品原则或验收方式，必须同步更新本规范。

必须触发反向提炼的情况：

- 既有 PRD 使用了稳定且可复用的章节，例如产品反馈同步规则、技术路线决策、UI 生成输入摘要。
- 既有 PRD 包含 EvoCraft 专用原则，例如应用集合定位、错题本长期入口、原图和干净题面并存。
- 既有 PRD 包含 AI/OCR、隐私、供应商、成本或模型分层边界，而规范没有明确检查点。
- 既有 PRD 的验收方式已经证明有复用价值，例如本地存储失败、删除/清空、人工复核、候选框恢复路径。

同步要求：

- 在同一变更中更新 PRD、PRD 编写规范、AGENTS 规则、项目记忆、路线图进度和文档索引。
- 如变更体现新的产品原则，还要同步想法胶囊。
- 不能只让单个 PRD 变好，而让规范继续遗漏该类要求。

检查点：

- 本次 PRD 更新是否发现了规范未覆盖的章节或产品原则？
- 这些内容是否已抽象成可复用规则，而不是复制某个 PRD 的具体正文？
- AGENTS 是否已经要求后续 PRD 遵循本规范？

## 4. 需求语句写作规则

### 4.1 强度词

统一使用以下强度词：

- `必须`：P0 或强约束，不满足则不能验收。
- `应该`：默认要做，只有在明确权衡后才能降级。
- `可以`：允许或可选，不构成本轮验收阻断。
- `不应`：默认禁止，除非后续 PRD 更新说明原因。
- `不做`：本轮明确非目标。

检查点：

- 是否把所有愿望都写成了“必须”？
- “应该”是否有降级条件？
- “可以”是否不会被误解为必须交付？

### 4.2 单一性

每条需求只表达一个要求。

差：

```text
系统必须支持上传图片、识别题目、保存记录，并在失败时提示用户。
```

好：

```text
PRD-FR-001：系统必须允许用户上传一张图片。
PRD-FR-002：系统必须在上传成功后展示图片预览。
PRD-FR-003：系统必须在识别失败时展示可继续操作的失败状态。
```

检查点：

- 是否能单独验证这一条？
- 如果实现失败，能否准确定位是哪条需求没满足？

### 4.3 可验证性

每条关键需求都要能通过至少一种方式验证：

- 测试：自动化或手动测试。
- 检查：查看界面、数据、配置、代码或文档。
- 演示：通过用户流程演示证明。
- 分析：通过日志、指标、模型调用记录或成本计算证明。

检查点：

- 是否知道用什么证据证明这条需求完成？
- 是否存在“体验更自然”“AI 更聪明”这种无法验证的需求？

### 4.4 可追踪性

每条 P0/P1 需求应能追溯到以下至少一种来源：

- 用户故事。
- 产品目标。
- 想法胶囊中的产品原则。
- 已确认决策。
- 风险或合规约束。
- 现有 PRD 的范围更新。

检查点：

- 如果删除这条需求，会破坏哪个目标或原则？
- 如果找不到来源，它是不是实现偏好或范围膨胀？

### 4.5 不提前锁死实现

PRD 应说明产品结果和约束，不轻易指定内部实现。

可以指定实现的情况：

- 隐私、安全、成本、合规或供应商限制要求。
- 已有架构已确认，且不遵守会破坏产品一致性。
- 用户明确要求某项技术或文件形态。

检查点：

- 这条需求是否告诉工程“必须用某个技术”，但没有产品理由？
- 是否可以改写成用户结果或系统能力？

## 5. 优先级规则

使用 P0 到 P3：

| 优先级 | 含义 | 验收影响 |
| --- | --- | --- |
| P0 | 核心闭环必需 | 不满足则不能认为版本完成 |
| P1 | 关键体验或风险控制 | 默认进入本轮，缺失需记录风险 |
| P2 | 重要增强 | 可延期，但应有后续计划 |
| P3 | 机会项或低优先级优化 | 不进入本轮验收 |

检查点：

- P0 是否只包含核心闭环？
- 是否把所有“不错的想法”都塞进 P0/P1？
- P2/P3 是否有明确延期理由？

## 7. PRD 评审流程

### 7.1 自检

PRD 作者先完成以下动作：

1. 通读目标、非目标、核心流程和验收标准，确认没有互相矛盾。
2. 扫描占位词：`TBD`、`TODO`、`待定`、`待补`、`后续再说`。
3. 扫描模糊词：`简单`、`快速`、`智能`、`友好`、`适当`、`尽量`。
4. 检查 P0/P1 需求是否都有验收方式。
5. 检查是否需要更新想法胶囊和进度记录。

### 7.2 产品评审

关注问题：

- 是否解决了真实用户问题？
- 是否符合 EvoCraft 当前阶段？
- 目标和非目标是否合理？
- 是否存在范围膨胀？
- 成功指标是否反映用户价值？

### 7.3 设计评审

关注问题：

- 页面目的是否清楚？
- 信息架构是否支持核心流程？
- 状态和错误是否完整？
- 文案是否适合学生、家长和老师？
- AI 不确定性是否被正确表达？

### 7.4 工程评审

关注问题：

- 需求是否可实现、可拆分、可验证？
- 数据模型是否足够支撑功能？
- 是否存在未说明的技术依赖？
- AI、存储、外部服务是否有抽象边界？
- 风险是否影响实现顺序？

### 7.5 测试评审

关注问题：

- 验收标准能否转成测试用例？
- 是否覆盖主路径、异常路径、空状态、失败状态？
- 是否有数据保存和回看验证？
- 是否有隐私、AI 复核和成本相关验证？

## 9. 常见问题

### 9.1 PRD 要不要写技术方案？

PRD 可以写技术约束、供应商边界、数据要求和风险，但不应替代技术设计。只有当技术选择直接影响产品价值、隐私、成本或可用性时，才应写进 PRD。

### 9.2 用户故事和功能需求是否重复？

不重复。用户故事解释用户为什么需要某能力，功能需求定义系统必须做什么。用户故事帮助理解，功能需求帮助实现和验收。

### 9.3 PRD 是否必须很长？

不必须。PRD 的长度由风险和范围决定。小功能可以短，但仍然要有问题、目标、范围、验收和风险。MVP 或关键能力必须完整。

### 9.4 什么时候必须更新想法胶囊？

当新 PRD、PRD 重大更新、明确产品想法、用户指出缺失功能或产品边界变化出现时，必须更新想法胶囊。想法胶囊记录提炼后的产品观点，不复制 PRD 正文。

### 9.5 什么时候只更新进度记录？

如果只是新增流程规范、实现计划、验证结果、项目状态，不改变产品想法或 PRD 范围，通常只需要更新项目记忆和进度记录，不需要写想法胶囊。
