# EvoCraft 错题收集应用 MVP PRD v1.0

最后更新：2026-05-10

文档状态：第一版正式 PRD，可作为 UI 生成和后续实现计划的输入。

## 1. 产品定位

EvoCraft 是面向上海孩子的 AI 学习助手应用集合。它不是单一错题工具，而是一个可以逐步加入多个学习应用的产品壳：错题收集、背单词、复习计划、阅读辅助、学习奖励、虚拟人物装扮等能力都可以在未来进入这个集合。

第一版 MVP 只实现集合里的第一个可用应用：`错题收集`。

这个 MVP 要证明两件事：

- EvoCraft 可以以 `App Hub / 应用集合首页` 的形态承载多个学习场景。
- 一张带有书写痕迹的错题照片，可以被整理成可信、可编辑、可复习、可继续分析的干净错题记录。

## 2. 背景与问题

孩子的错题通常不是干净题面，而是包含学生答案、批改勾叉、圈画、涂改、老师评语、草稿笔迹等痕迹的真实照片。如果直接保存原图，后续复习时会暴露答案和错误痕迹，也不利于生成相似题、做错因分析或形成复习计划。

因此，第一版错题收集应用不只是“上传图片并 OCR”，而是要形成一个结构化错题记录：

- 原图保留，用于溯源、校验和重新处理。
- 干净题面生成，用于复习、分析和后续练习。
- AI 识别结果可编辑，避免把不确定识别当成最终结果。

## 3. 目标

- 保留 EvoCraft 应用集合的顶层结构，让后续学习应用可以自然加入。
- 提供一个可用的 `错题收集` 应用入口。
- 支持上传一张包含一道错题的图片。
- 支持语文、数学、英语三类科目。
- 识别题目文字、图形、公式、表格等关键内容。
- 生成或保存去除书写痕迹后的干净题面。
- 保留原图，避免去痕失败后无法恢复。
- 允许用户检查、修正、保存错题记录。
- 提供已保存错题详情页，默认展示干净题面。

## 4. 非目标

- 不做整张试卷的多题自动拆分。
- 不做完整错题分析引擎。
- 不做账号系统、云同步、班级管理或家长看板。
- 不做金币经济、虚拟人物装扮或奖励系统业务逻辑。
- 不实现背单词、阅读辅助、复习计划等后续应用。
- 不实现复杂插件系统；第一版只保留清晰的应用集合信息架构。
- 不做原生移动端应用。
- 不承诺 AI 识别或自动去痕在无人复核时完全正确。

## 5. 目标用户

主要用户：

- 上海学生，尤其是需要整理错题、后续复习的孩子。

辅助用户：

- 帮助孩子上传、检查、修正识别结果的家长或老师。

第一版操作假设：

- 桌面端优先。
- 允许家长或老师协助操作。
- UI 文案应让学生也能理解，不应只面向成人管理者。

## 6. MVP 默认决策

这些是第一版 PRD 为了推进 UI 和开发而采用的保守默认值：

- 存储默认本地优先；不把账号和云同步作为 MVP 前置条件。
- UI 生成阶段可以使用 mock 识别结果，不要求立即接入真实 OCR 或多模态模型。
- 后续真实接入外部 AI 服务前，必须明确儿童学习照片的上传、存储、删除、授权和隐私边界。
- App Hub 第一版只需要一个轻量入口；未来应用可以低优先级展示为“规划中”，但不能干扰错题收集主流程。
- 第一版不要求全自动完美去痕；允许出现“需复核”状态。

## 7. 决策边界

开发过程中可以直接决定：

- UI 组件拆分、页面路由命名、局部交互细节。
- mock 数据结构和本地状态管理方式。
- 识别过程的前端状态表达，例如处理中、需复核、失败。
- 未来应用占位入口的低干扰展示方式。

需要再次确认后才能决定：

- 是否上传真实儿童学习照片到外部 AI/OCR 服务。
- 是否引入账号体系、云同步或后端数据库。
- 是否扩大到多题拆分、错因分析、相似题生成。
- 是否加入金币、虚拟人物、装扮等游戏化业务逻辑。
- 是否开始移动端原生实现。

## 8. 核心用户故事

作为学生或家长，我打开 EvoCraft 后能看到它是一个 AI 学习助手应用集合；第一版我进入“错题收集”应用，上传一张错题照片，让 EvoCraft 帮我识别题目内容、去除书写痕迹、保留原图，并生成一个可以复习的干净错题记录。

## 9. 核心流程

1. 用户打开 EvoCraft。
2. 用户看到 `App Hub / 应用集合首页`，当前可用应用是“错题收集”。
3. 用户进入“错题收集”应用。
4. 用户上传或拖入一张图片。
5. 系统展示图片预览。
6. 系统进入识别流程，产出：
   - 科目建议；
   - 题目文字草稿；
   - 原图引用；
   - 题目视觉片段；
   - 干净题面；
   - 识别状态；
   - 去痕状态。
7. 用户检查并编辑字段。
8. 用户保存错题记录。
9. 系统展示已保存错题详情，默认展示干净题面，同时提供查看原图入口。

## 10. 页面与信息架构

### 10.1 App Hub / 应用集合首页

目的：让用户理解 EvoCraft 是 AI 学习助手应用集合，同时快速进入第一个可用应用。

必须包含：

- EvoCraft 名称。
- 简短定位：AI 学习助手应用集合。
- `错题收集` 应用入口，标记为当前可用。
- 未来应用区域，例如背单词、复习计划、学习奖励，标记为规划中或弱化展示。

设计要求：

- 不做营销式落地页。
- 第一屏应是可操作工具入口。
- 视觉焦点放在“错题收集”。
- 后续应用入口不能抢占主流程注意力。

### 10.2 错题收集上传页

目的：让用户上传一张错题图片，并确认当前处理对象。

必须包含：

- 图片上传区，支持点击和拖拽。
- 图片预览。
- 支持替换图片。
- 科目选择：语文、数学、英语。
- 进入识别/整理的主操作按钮。

错误状态：

- 文件格式不支持。
- 文件为空或损坏。
- 图片过大。
- 用户取消上传。

### 10.3 识别检查页

目的：让用户在保存前确认 AI 识别结果和去痕结果。

桌面布局：

- 左侧展示原图、题目截图或干净题面预览。
- 右侧展示可编辑字段。
- 提供原图/干净题面切换。

必须包含：

- 科目字段。
- 题目标题或短标签。
- 题目文字。
- 学生答案字段。
- 正确答案字段。
- 备注字段。
- 识别状态。
- 去痕状态。
- 保存按钮。

### 10.4 已保存错题详情页

目的：展示一个可复习、可继续分析的错题记录。

必须包含：

- 科目。
- 标题。
- 干净题面，作为默认视觉内容。
- 查看原图入口。
- 题目文字。
- 学生答案、正确答案、备注。
- 创建时间。
- 后续分析入口的占位位置，但不实现分析功能。

## 11. 功能需求

### 11.1 应用集合壳

- EvoCraft 第一版应保留应用集合首页或等价顶层入口。
- `错题收集` 是第一版唯一可用应用。
- 顶层结构应方便后续加入背单词、阅读辅助、复习计划、学习奖励等应用。
- 不需要复杂插件系统或动态应用市场。

### 11.2 图片上传

- 一次只接受一张图片。
- 支持 JPG、PNG、HEIC 的产品设计；技术实现如暂不支持 HEIC，界面应给出清晰提示。
- 上传后展示预览。
- 保存前允许替换图片。
- 原图必须保留为记录的一部分。

### 11.3 科目

- 支持语文、数学、英语。
- 系统可以给出科目建议。
- 用户必须可以手动覆盖科目。
- 识别不确定时，不应假装确定。

### 11.4 识别

- 识别结果是草稿，不是最终答案。
- 尽可能提取题目文字。
- 对图形、几何图、表格、公式、低置信度区域保留视觉内容。
- 识别内容以可编辑字段展示。
- 识别失败时，用户仍可手动填写并保存。

### 11.5 书写痕迹处理

- 系统应生成一个“干净题面”，用于复习、分析和生成相似题。
- 干净题面应尽量去除或隔离学生答案、批改勾叉、老师评语、圈画、涂改、草稿笔迹。
- 干净题面必须保留题干、选项、公式、图形、表格、几何辅助线等题目必要内容。
- 原图不能被干净题面覆盖。
- 当系统不确定某个笔迹是否属于题目本身时，应保守保留并提示复核。
- 学生答案可作为单独字段保存，但不应默认混在干净题面里。

### 11.6 记录编辑

用户可以编辑：

- 科目。
- 标题。
- 题目文字。
- 学生答案。
- 正确答案。
- 备注。

用户不需要编辑：

- 唯一记录 id。
- 创建时间。
- 系统内部状态字段。

### 11.7 保存

保存后的错题记录必须包含：

- 记录 id。
- 所属应用 id。
- 创建时间。
- 更新时间。
- 科目。
- 标题。
- 题目文字。
- 原图引用。
- 干净题面引用。
- 识别状态。
- 去痕状态。

可选包含：

- 题目视觉片段引用。
- 学生答案。
- 正确答案。
- 备注。
- 识别置信度。
- 去痕置信度。

## 12. 数据模型草案

```ts
type LearningAppId = "wrong_question_capture";

type Subject = "chinese" | "math" | "english";

type ProcessingStatus =
  | "not_started"
  | "processing"
  | "needs_review"
  | "reviewed"
  | "failed";

type WrongQuestionRecord = {
  id: string;
  appId: LearningAppId;
  createdAt: string;
  updatedAt: string;
  subject: Subject;
  title: string;
  questionText: string;
  originalImageUri: string;
  cleanedQuestionImageUri?: string;
  visualSnippetUri?: string;
  studentAnswer?: string;
  correctAnswer?: string;
  notes?: string;
  recognitionStatus: ProcessingStatus;
  recognitionConfidence?: number;
  cleanupStatus: ProcessingStatus;
  cleanupConfidence?: number;
};
```

## 13. AI 能力契约

第一版 UI 和产品流程不绑定具体模型供应商，但需要定义 AI 能力的输入输出。

输入：

- 一张错题图片。
- 用户选择或系统建议的科目。

输出：

- 科目建议。
- 题目文字草稿。
- 学生答案草稿。
- 正确答案草稿。
- 干净题面图片。
- 识别状态。
- 去痕状态。
- 置信度或需复核标记。

原则：

- 原图优先保留。
- 干净题面用于复习。
- 低置信度结果必须可编辑。
- 去痕策略必须保守，避免误删题目本身。

## 14. 隐私与安全

上传图片可能包含姓名、学校、班级、字迹、分数、老师批注等敏感信息。MVP 应采用最小收集原则。

第一版约束：

- 不要求账号系统。
- 不要求云同步。
- 不默认把真实儿童学习照片发送到外部服务。
- 如后续接入外部 AI 服务，必须先明确上传提示、授权方式、存储位置、保留周期和删除机制。

## 15. UI 文案与语气

语言：

- 中文优先。
- 面向学生和家长都能理解。

语气：

- 清楚。
- 温和。
- 不恐吓用户。
- 不把 AI 结果包装成绝对正确。

避免：

- 在可见 UI 中大量使用 OCR、多模态、置信度等技术词。
- 让孩子直接面对复杂参数。
- 让未来应用占位喧宾夺主。

## 16. 成功指标

产品层面：

- 用户可以从 App Hub 进入错题收集应用。
- 用户可以完成一次上传、检查、保存、查看详情闭环。
- 保存记录默认可用于复习，而不是只保存一张带答案痕迹的照片。

体验层面：

- 用户能明确区分原图和干净题面。
- 用户能理解识别结果需要复核。
- 用户在识别失败时仍能手动保存。

后续规划层面：

- 文档和 UI 不把 EvoCraft 锁死为单一错题工具。
- 后续应用能自然接入 App Hub。

## 17. 验收标准

- App Hub 展示 EvoCraft 名称和 AI 学习助手应用集合定位。
- App Hub 中 `错题收集` 是当前可用应用。
- 用户可以上传一张图片并看到预览。
- 用户可以选择语文、数学或英语。
- 系统可以展示识别草稿，或展示清晰的识别失败状态。
- 系统可以展示干净题面，或展示需复核/失败状态。
- 用户可以在保存前编辑识别字段。
- 用户可以保存一个错题记录。
- 用户可以打开已保存错题详情页。
- 详情页默认展示干净题面。
- 详情页提供查看原图入口。
- 原图保留在记录中。
- 完成 MVP 不依赖账号、云同步、游戏化、多题拆分或其他学习应用。

## 18. 后续决策点

- 第一版 App Hub 是否在 UI 中展示未来应用占位，还是只展示当前可用应用。
- 第一版技术栈选择。
- 第一版存储采用浏览器本地存储、桌面本地文件，还是轻量数据库。
- 识别能力先 mock，还是直接接入真实 AI/OCR 服务。
- 书写痕迹去除采用自动处理、人工框选辅助，还是先用 mock 结果完成 UI 闭环。
- 第一批目标年级是否聚焦小学高年级到初中。

## 19. UI 生成输入摘要

生成 UI 图时，应优先体现：

- EvoCraft 是应用集合，不是单一错题工具。
- 第一版只有 `错题收集` 可用。
- 桌面端双栏检查体验。
- 原图和干净题面的区别。
- AI 结果是草稿，需要用户复核。
- 保存后的错题记录清晰、安静、适合复习。
