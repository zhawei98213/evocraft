# EvoCraft 错题收集应用 MVP PRD

最后更新：2026-05-10

## 1. 概述

EvoCraft 的产品本体是面向上海孩子的 AI 学习助手应用集合。第一阶段是桌面优先 MVP，只实现集合里的第一个应用：`错题收集`。这个 MVP 聚焦一个完整闭环：用户上传一张错题照片，系统识别并整理为一个结构化错题记录，为后续分析、复习和游戏化成长体系打基础。

这个 MVP 要证明两件事：第一，EvoCraft 可以作为学习应用集合承载多个学习场景；第二，拍下来的错题可以变成可信、可编辑、可保存、可继续使用的学习资料。它不试图一次性完成整个学习助手愿景。

## 2. 目标

- 保留 EvoCraft 应用集合的顶层结构，让后续学习应用可以自然加入。
- 用户可以上传一张包含一道错题的照片。
- 系统可以识别照片中的基础题目信息。
- 第一批支持科目：语文、数学、英语。
- 对图形、表格、几何图、公式、低置信度区域保留视觉内容，不强行转成不可靠文本。
- 对学生书写、批改勾叉、圈画、涂改等痕迹进行去除或隔离，生成适合复习的干净题面。
- 用户可以在保存前检查并编辑 AI 识别字段。
- 用户可以保存一个错题记录，并在详情页查看。

## 3. 非目标

- 不做整张试卷的多题自动拆分。
- 不做完整学习分析引擎。
- 不做账号系统、云同步、班级管理、家长看板。
- 不做金币经济或虚拟人物装扮。
- 不实现背单词、阅读辅助、复习计划等后续应用。
- 不实现复杂插件系统；第一版只保留清晰的应用集合信息架构。
- 不做原生移动端应用。
- 不承诺 AI 识别结果无需人工检查即可完全正确。
- 不承诺第一版自动去痕能处理所有复杂书写场景；必须保留原图，并允许用户复核。

## 4. 目标用户

主要用户：

- 想整理错题、便于后续复习的上海学生。

辅助用户：

- 帮助孩子上传、检查、修正识别结果的家长或老师。

MVP 假设：

- 第一版桌面端可能更多由成人协助操作，所以 UI 要适合仔细检查和修正。

## 5. 用户故事

作为学生或家长，我打开 EvoCraft 后能看到它是一个 AI 学习助手应用集合；第一版我进入“错题收集”应用，上传一张错题照片，让 EvoCraft 帮我识别科目和题目内容；我可以修正 AI 识别错误，然后保存为错题记录，方便以后复习或分析。

## 6. MVP 流程

1. 用户打开 EvoCraft。
2. 用户看到 EvoCraft 应用集合首页，当前可用应用是“错题收集”。
3. 用户进入“错题收集”应用。
4. 用户上传或拖入一张图片。
5. 应用展示图片预览。
6. 应用运行识别并返回：
   - 识别出的科目；
   - 题目文字；
   - 可能的学生答案区域；
   - 图片中可见的答案、解析或备注；
   - 原图引用、题目视觉片段、去除书写痕迹后的干净题面。
7. 用户检查并编辑识别字段。
8. 用户保存错题记录。
9. 应用展示已保存的错题详情。

## 7. 功能需求

### 应用集合壳

- EvoCraft 第一版应保留应用集合首页或等价的顶层入口。
- 顶层入口应明确 EvoCraft 是 AI 学习助手应用集合，而不是单一错题工具。
- `错题收集` 是第一版唯一可用应用。
- 未来应用可以在文档、路线图或低干扰占位入口中出现，但不能干扰错题收集主流程。
- 顶层结构应方便后续加入背单词、阅读辅助、复习计划、学习奖励等应用。

### 图片上传

- 一次只接受一张图片。
- 支持常见图片格式，例如 JPG、PNG。
- 上传后显示清晰预览。
- 保存前允许用户替换图片。

### 科目

- 支持科目：语文、数学、英语。
- 系统可以自动识别科目，但用户必须可以手动覆盖。
- 识别不确定时，优先提示用户选择，而不是假装确定。

### 识别

- 尽可能提取题目文字。
- 对图形、几何图、表格、公式、手写内容、低置信度区域保留原始视觉内容。
- 识别内容以可编辑字段展示。
- 识别不确定性要可见，但不要造成紧张感。

### 书写痕迹处理

- 真实错题照片通常包含学生作答、批改勾叉、老师评语、圈画、涂改、草稿笔迹等痕迹。
- 系统应生成一个“干净题面”，用于后续复习、分析和生成相似题。
- 干净题面应尽量去除或隔离非题目本身的书写痕迹，同时保留题干、选项、公式、图形、表格、几何辅助线等题目必要内容。
- 原图必须保留，不能被干净题面覆盖。
- 如果自动去痕不确定，界面应提示用户复核，避免误删题目内容。
- 如果学生答案对错因分析有价值，可将其作为单独字段保存，但不应默认混在干净题面里。

### 记录编辑

用户可以编辑：

- 科目。
- 题目标题或短标签。
- 题目文字。
- 学生答案，如果图片中可见或用户手动输入。
- 正确答案，如果图片中可见或用户手动输入。
- 备注。

### 保存

保存后的错题记录必须包含：

- 唯一记录 id。
- 创建时间。
- 科目。
- 题目文字。
- 原图引用。
- 可选的题目视觉片段引用。
- 去除书写痕迹后的干净题面引用。
- 去痕状态，例如未处理、处理中、需复核、已复核、失败。
- 可选的学生答案。
- 可选的正确答案。
- 可选备注。
- 识别状态或识别置信度。

### 详情页

- 用清晰版式展示已保存题目。
- 展示科目。
- 默认展示干净题面，并提供查看原图的入口。
- 展示识别并编辑后的文字。
- 明确表现“已保存，可用于后续分析”。

## 8. 第一版 UI 需求

第一版 UI 桌面优先，重点支持仔细检查。

核心页面：

- App Hub / 应用集合首页：展示 EvoCraft 名称、AI 学习助手应用集合定位，以及“错题收集”入口。
- 上传页：大上传区、科目选择、照片预览。
- 识别检查页：左侧图片，右侧可编辑字段。
- 已保存详情页：以干净的阅读版式展示最终错题记录，默认展示去痕后的题面。

布局要求：

- 桌面宽屏下优先使用双栏检查布局。
- 用户编辑字段时，照片应保持可见。
- 不做营销式落地页；第一屏应该像一个真实可用的学习工具。
- 第一屏要能看出 EvoCraft 后续可扩展为多个学习应用，但视觉焦点仍放在当前可用的“错题收集”。
- 视觉可以有面向孩子的温暖感，但不能玩具化。
- 为后续游戏元素预留气质和空间，但 MVP 流程内不放金币和装扮系统。

文案语气：

- 清楚、温和、鼓励。
- 使用学生和家长都能理解的中文标签。
- 可见 UI 文案避免出现过多 OCR、置信度、多模态等技术词。

## 9. 数据模型草案

```ts
type Subject = "chinese" | "math" | "english";

type RecognitionStatus = "not_started" | "processing" | "needs_review" | "reviewed" | "failed";

type CleanupStatus = "not_started" | "processing" | "needs_review" | "reviewed" | "failed";

type LearningAppId = "wrong_question_capture";

type WrongQuestionRecord = {
  id: string;
  createdAt: string;
  updatedAt: string;
  appId: LearningAppId;
  subject: Subject;
  title: string;
  questionText: string;
  studentAnswer?: string;
  correctAnswer?: string;
  notes?: string;
  originalImageUri: string;
  visualSnippetUri?: string;
  cleanedQuestionImageUri?: string;
  recognitionStatus: RecognitionStatus;
  recognitionConfidence?: number;
  cleanupStatus: CleanupStatus;
  cleanupConfidence?: number;
};
```

## 10. AI 识别说明

MVP 中 AI 输出必须被视为“草稿”，而不是最终答案。

识别优先级：

1. 保留上传原图。
2. 生成去除书写痕迹后的干净题面。
3. 尽量提取有用文字。
4. 让不确定结果可编辑。
5. 避免用假的确定性误导用户。

图形题即使文字提取不完整，也应该因为保留了图片内容而仍然可用。

书写痕迹去除必须保守：当模型无法判断某条笔迹是学生作答还是题目本身的一部分时，应保留内容并提示复核，而不是直接删除。

## 11. 隐私与安全说明

产品面向孩子，照片处理必须谨慎设计。

MVP 设计应避免收集不必要的个人信息。上传照片可能包含姓名、学校、班级、字迹、分数等信息。后续只要引入云端上传，就必须先明确存储位置、保留周期、删除机制、授权与同意流程。

## 12. 验收标准

- 用户可以上传一张图片并看到预览。
- 用户可以从 EvoCraft 应用集合首页进入“错题收集”应用。
- 用户可以选择语文、数学或英语。
- 用户可以看到 AI 识别草稿，或看到清晰的识别失败状态。
- 用户可以编辑识别字段。
- 用户可以保存一个错题记录。
- 用户可以打开已保存错题的详情页。
- 原图会保留在错题记录中。
- 已保存记录默认展示去除书写痕迹后的干净题面；当去痕失败或需复核时，界面有明确状态。
- 完成 MVP 流程不依赖游戏化、账号系统、多题拆分。

## 13. 开放问题

- 第一版实现时，错题记录存浏览器本地、系统本地文件，还是小型数据库？
- App Hub 第一版是否展示未来应用占位入口，还是只在文档和导航结构中保留？
- UI 开发阶段先 mock 识别结果，还是直接接入 AI/OCR 服务？
- 书写痕迹去除第一版采用自动处理、人工框选辅助，还是先 mock 去痕结果来完成 UI 闭环？
- 第一版 UI 文案主要面向哪个年级段？
- 第一版操作对象主要是家长、孩子，还是明确支持两者共同使用？
- 后续用哪个 UI 生成工具或格式消费这份 PRD？
