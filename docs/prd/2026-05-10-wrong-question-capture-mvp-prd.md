# EvoCraft 错题收集应用 MVP PRD v1.3

最后更新：2026-05-15

文档状态：v1.3，已补充多题照片中的单题区域选择流程、候选框删除逻辑，以及真实 AI 接入时采用国内模型优先、分层调用、provider-agnostic 的模型策略。

## 1. 产品定位

EvoCraft 是面向上海孩子的 AI 学习助手应用集合。它不是单一错题工具，而是一个可以逐步加入多个学习应用的产品壳：错题收集、背单词、复习计划、阅读辅助、学习奖励、虚拟人物装扮等能力都可以在未来进入这个集合。

第一版 MVP 只实现集合里的第一个可用应用：`错题收集`。

这个 MVP 要证明两件事：

- EvoCraft 可以以 `App Hub / 应用集合首页` 的形态承载多个学习场景。
- 一张带有书写痕迹的错题照片，可以被整理成可信、可编辑、可复习、可继续分析的干净错题记录。

## 2. 背景与问题

孩子的错题通常不是干净题面，而是包含学生答案、批改勾叉、圈画、涂改、老师评语、草稿笔迹等痕迹的真实照片。如果直接保存原图，后续复习时会暴露答案和错误痕迹，也不利于生成相似题、做错因分析或形成复习计划。

因此，第一版错题收集应用不只是“上传图片并 OCR”，而是要形成一个结构化错题记录：

- 原图保留，用于溯源、校验和重新处理。
- 干净题面生成，用于复习、分析和后续练习。
- AI 识别结果可编辑，避免把不确定识别当成最终结果。

## 3. 目标

- 保留 EvoCraft 应用集合的顶层结构，让后续学习应用可以自然加入。
- 提供一个可用的 `错题收集` 应用入口。
- 支持上传一张可能包含多道题的图片，并在识别前确认本次要处理的一道题区域。
- 支持语文、数学、英语三类科目。
- 识别题目文字、图形、公式、表格等关键内容。
- 生成或保存去除书写痕迹后的干净题面。
- 同时保留整张原图和用户确认后的题目区域，避免区域识别或去痕失败后无法恢复。
- 允许用户检查、修正、保存错题记录。
- 提供 `错题本` 列表入口，让用户保存后可以查看全部已保存记录。
- 提供已保存错题详情页，默认展示干净题面。

## 4. 非目标

- 不做整张试卷的多题自动拆分。
- 不做一次保存多道题、连续批量拆题或多框合并。
- 不做完整错题分析引擎。
- 不做账号系统、云同步、班级管理或家长看板。
- 不做金币经济、虚拟人物装扮或奖励系统业务逻辑。
- 不实现背单词、阅读辅助、复习计划等后续应用。
- 不实现复杂插件系统；第一版只保留清晰的应用集合信息架构。
- 不做原生移动端应用。
- 不承诺 AI 识别或自动去痕在无人复核时完全正确。
- 不把 GPT-5.5 或海外旗舰模型作为默认运行时投入；真实接入优先采用国内可访问、性价比更高的视觉/OCR 模型。

## 5. 目标用户

主要用户：

- 上海学生，尤其是需要整理错题、后续复习的孩子。

辅助用户：

- 帮助孩子上传、检查、修正识别结果的家长或老师。

第一版操作假设：

- 桌面端优先。
- 允许家长或老师协助操作。
- UI 文案应让学生也能理解，不应只面向成人管理者。

## 6. MVP 默认决策

这些是第一版 PRD 为了推进 UI 和开发而采用的保守默认值：

- 存储默认本地优先；不把账号和云同步作为 MVP 前置条件。
- UI 生成阶段可以使用 mock 识别结果，不要求立即接入真实 OCR 或多模态模型。
- 真实 AI/OCR 接入时，默认采用国内模型优先；第一候选为阿里云百炼 Qwen 体系，保留火山引擎豆包视觉理解等备选。
- 模型调用必须分层：先用低成本视觉 OCR/轻量结构化模型处理常规题，只有低置信度、复杂公式或几何图时才升级到更强模型。
- 后续真实接入外部 AI 服务前，必须明确儿童学习照片的上传、存储、删除、授权和隐私边界。
- App Hub 第一版只需要一个轻量入口；未来应用可以低优先级展示为“规划中”，但不能干扰错题收集主流程。
- 第一版不要求全自动完美去痕；允许出现“需复核”状态。

## 7. 决策边界

开发过程中可以直接决定：

- UI 组件拆分、页面路由命名、局部交互细节。
- mock 数据结构和本地状态管理方式。
- 识别过程的前端状态表达，例如处理中、需复核、失败。
- 国内模型供应商的适配层接口命名和 mock/fallback 策略，只要不破坏 provider-agnostic 契约。
- 未来应用占位入口的低干扰展示方式。

需要再次确认后才能决定：

- 是否上传真实儿童学习照片到外部 AI/OCR 服务。
- 是否引入账号体系、云同步或后端数据库。
- 是否扩大到多题拆分、错因分析、相似题生成。
- 是否从第一候选 Qwen 体系切换为其他国内供应商，或引入多供应商并行 A/B。
- 是否加入金币、虚拟人物、装扮等游戏化业务逻辑。
- 是否开始移动端原生实现。

## 8. 核心用户故事

作为学生或家长，我打开 EvoCraft 后能看到它是一个 AI 学习助手应用集合；第一版我进入“错题收集”应用，上传一张可能包含多道题的错题照片，让 EvoCraft 先帮我找出候选题目区域，我确认或手动画出本次要整理的一道题后，再识别题目内容、去除书写痕迹、保留整张原图，并生成一个可以复习的干净错题记录。保存后，我可以从明确的 `错题本` 入口查看全部已保存错题，再打开单条详情复习干净题面、题目区域或整张原图。

## 9. 核心流程

1. 用户打开 EvoCraft。
2. 用户看到 `App Hub / 应用集合首页`，当前可用应用是“错题收集”。
3. 用户进入“错题收集”应用。
4. 用户上传或拖入一张图片。
5. 系统展示图片预览。
6. 系统进入 `选择题目区域` 步骤：
   - AI 自动标出候选题目区域；
   - 用户选择一个候选框；
   - 用户可拖动和缩放边界微调；
   - 用户可删除误识别或干扰后续手动画框的候选框；
   - 如果 AI 没框中目标题，用户可进入 `手动画框` 模式；
   - 用户确认本次要识别的一道题。
7. 系统只针对确认后的题目区域进入识别流程，产出：
   - 科目建议；
   - 题目文字草稿；
   - 原图引用；
   - 题目区域引用；
   - 题目视觉片段；
   - 干净题面；
   - 识别状态；
   - 去痕状态。
8. 用户检查并编辑字段。
9. 用户保存错题记录。
10. 系统将错题加入 `错题本` 列表，并允许用户从列表查看已保存记录。
11. 用户打开某条已保存错题详情。
12. 系统默认展示干净题面，同时提供查看题目区域和整张原图入口。

## 10. 页面与信息架构

### 10.1 App Hub / 应用集合首页

目的：让用户理解 EvoCraft 是 AI 学习助手应用集合，同时快速进入第一个可用应用。

必须包含：

- EvoCraft 名称。
- 简短定位：AI 学习助手应用集合。
- `错题收集` 应用入口，标记为当前可用。
- 未来应用区域，例如背单词、复习计划、学习奖励，标记为规划中或弱化展示。

设计要求：

- 不做营销式落地页。
- 第一屏应是可操作工具入口。
- 视觉焦点放在“错题收集”。
- 后续应用入口不能抢占主流程注意力。

### 10.2 错题收集上传页

目的：让用户上传一张错题图片，并进入后续选题区域流程。

必须包含：

- 图片上传区，支持点击和拖拽。
- 图片预览。
- 支持替换图片。
- 科目选择：语文、数学、英语。
- 进入 `选择题目区域` 的主操作按钮。

错误状态：

- 文件格式不支持。
- 文件为空或损坏。
- 图片过大。
- 用户取消上传。

### 10.3 选择题目区域页

目的：从一张可能包含多道题的照片里，确认本次要识别和整理的一道题。

必须包含：

- 整张原图画布。
- AI 自动识别出的候选题目区域框。
- 当前选中区域的高亮框。
- 候选区域列表或等价切换控件。
- 删除候选框入口，用于移除误识别区域或避免与手动画框重叠造成混乱。
- `手动画框` 入口，用于 AI 没框中目标题时从空白开始框选。
- 拖动和缩放边界的基础能力。
- 缩放查看能力，至少支持放大、缩小和适配画布。
- `重新自动找题` 或等价重试入口。
- `确认此区域并识别` 主操作。
- 返回上传页入口。

设计要求：

- 这是独立步骤，不是上传页里的临时裁剪弹窗。
- 第一版只确认一道题，不进入批量拆题。
- 若 AI 没有候选结果，页面应直接引导用户手动画框。
- 删除候选框后不应影响整张原图，用户仍可手动画框或重新自动找题。
- 用户确认后，后续识别和去痕只处理确认后的题目区域。
- 保存记录时必须同时保留整张原图和确认后的题目区域。

明确不做：

- 不做一次保存多道题。
- 不做多框合并。
- 不做复杂标注历史、撤销重做或高级图像标注工具。

### 10.4 识别检查页

目的：让用户在保存前确认 AI 识别结果和去痕结果。

桌面布局：

- 左侧展示原图、题目截图或干净题面预览。
- 右侧展示可编辑字段。
- 提供原图/干净题面切换。

必须包含：

- 科目字段。
- 题目标题或短标签。
- 题目文字。
- 学生答案字段。
- 正确答案字段。
- 备注字段。
- 识别状态。
- 去痕状态。
- 保存按钮。

### 10.5 错题本 / 已保存错题列表

目的：让用户知道收集后的错题沉淀在哪里，并能回到全部已保存记录。

必须包含：

- 左侧导航或等价一级入口中的 `错题本`。
- 全部已保存错题列表。
- 已保存数量或空状态提示。
- 每条记录展示科目、标题、保存时间和去痕/复核状态。
- 每条记录可打开详情页。
- 空状态应提示用户继续收集第一道错题。
- 提供继续收集或新增错题入口。

设计要求：

- `错题本` 不应藏在上传页的最近记录里。
- 列表默认强调干净题面可用于复习，同时保留原图可追溯。
- 上传流程、错题本列表和详情页应形成清晰闭环：收集 -> 列表 -> 详情 -> 返回列表。

### 10.6 已保存错题详情页

目的：展示一个可复习、可继续分析的错题记录。

必须包含：

- 科目。
- 标题。
- 干净题面，作为默认视觉内容。
- 查看确认后的题目区域入口。
- 查看整张原图入口。
- 题目文字。
- 学生答案、正确答案、备注。
- 创建时间。
- 后续分析入口的占位位置，但不实现分析功能。

## 11. 功能需求

### 11.1 应用集合壳

- EvoCraft 第一版应保留应用集合首页或等价顶层入口。
- `错题收集` 是第一版唯一可用应用。
- 顶层结构应方便后续加入背单词、阅读辅助、复习计划、学习奖励等应用。
- 不需要复杂插件系统或动态应用市场。

### 11.2 图片上传

- 一次只接受一张图片。
- 支持 JPG、PNG、HEIC 的产品设计；技术实现如暂不支持 HEIC，界面应给出清晰提示。
- 上传后展示预览。
- 保存前允许替换图片。
- 原图必须保留为记录的一部分。

### 11.3 题目区域选择

- 上传后必须进入独立 `选择题目区域` 步骤。
- 系统应尝试自动识别照片中的候选题目区域。
- 用户必须可以选择一个候选区域作为本次识别目标。
- 用户必须可以拖动、缩放候选框边界，修正 AI 框选结果。
- 用户必须可以删除 AI 误识别的候选框，避免候选框与手动画框重叠或干扰选择。
- 删除当前选中区域后，系统应自动选择剩余候选框；如果没有剩余候选框，应进入未选择状态，并引导用户手动画框或重新自动找题。
- 如果 AI 没有框中目标题，用户必须可以使用 `手动画框` 从空白开始框出一道题。
- 用户确认前不应开始最终题目识别和去痕。
- 用户确认后，识别、结构化整理、干净题面生成都以确认后的题目区域为主输入。
- 保存记录时必须保留整张原图和确认后的题目区域。
- 如果自动找题失败，页面应进入可继续的手动画框状态，而不是阻断流程。

### 11.4 科目

- 支持语文、数学、英语。
- 系统可以给出科目建议。
- 用户必须可以手动覆盖科目。
- 识别不确定时，不应假装确定。

### 11.5 识别

- 识别结果是草稿，不是最终答案。
- 尽可能提取题目文字。
- 对图形、几何图、表格、公式、低置信度区域保留视觉内容。
- 识别内容以可编辑字段展示。
- 识别失败时，用户仍可手动填写并保存。
- 真实模型接入时，默认使用国内可访问模型；不把 GPT-5.5 作为常规识别链路。
- 常规链路优先采用阿里云百炼 Qwen 体系：视觉 OCR 模型处理题目区域，轻量模型负责结构化整理。
- 复杂公式、几何图、低置信度或 OCR 冲突时，再升级到更强的国内模型。
- 供应商实现必须通过 provider-agnostic 适配层接入，避免业务 UI 绑定某个模型 SDK。

### 11.6 书写痕迹处理

- 系统应生成一个“干净题面”，用于复习、分析和生成相似题。
- 干净题面应尽量去除或隔离学生答案、批改勾叉、老师评语、圈画、涂改、草稿笔迹。
- 干净题面必须保留题干、选项、公式、图形、表格、几何辅助线等题目必要内容。
- 原图不能被干净题面覆盖。
- 当系统不确定某个笔迹是否属于题目本身时，应保守保留并提示复核。
- 学生答案可作为单独字段保存，但不应默认混在干净题面里。
- 第一版不依赖大模型直接生成不可编辑图片作为唯一干净题面；优先用结构化识别结果和保留的图形片段重新渲染可复核题面。

### 11.7 记录编辑

用户可以编辑：

- 科目。
- 标题。
- 题目文字。
- 学生答案。
- 正确答案。
- 备注。

用户不需要编辑：

- 唯一记录 id。
- 创建时间。
- 系统内部状态字段。

### 11.8 保存

保存后的错题记录必须包含：

- 记录 id。
- 所属应用 id。
- 创建时间。
- 更新时间。
- 科目。
- 标题。
- 题目文字。
- 原图引用。
- 题目区域引用。
- 干净题面引用。
- 题目区域坐标。
- 识别状态。
- 去痕状态。

可选包含：

- 题目视觉片段引用。
- 学生答案。
- 正确答案。
- 备注。
- 识别置信度。
- 去痕置信度。
- 区域识别置信度。
- 模型供应商和模型 id，用于后续排查和重新处理。

### 11.9 错题本查看

- 第一版必须提供独立 `错题本` 或等价一级入口。
- `错题本` 展示全部已保存错题，不能只展示最近一条记录。
- 用户保存错题后，必须能从 `错题本` 找回该记录。
- `查看全部`、`查看更多` 等入口必须进入错题本列表，而不是直接跳到某一条详情。
- 详情页必须提供返回 `错题本` 的路径。

### 11.10 产品反馈同步规则

- 当用户明确指出“缺少某功能”“补某功能”“这个功能在哪里”等产品缺口时，相关 PRD 必须在同一轮变更中更新。
- 如果该反馈反映新的产品原则、信息架构、用户价值或流程判断，必须同步更新想法胶囊。
- 代码实现、PRD 和想法胶囊应在同一提交范围内保持一致。

## 13. AI 能力契约

第一版 UI 和产品流程不绑定具体模型供应商，但需要定义 AI 能力的输入输出。

输入：

- 一张可能包含多道题的错题图片。
- 用户确认后的题目区域。
- 用户选择或系统建议的科目。

输出：

- 候选题目区域。
- 科目建议。
- 题目文字草稿。
- 学生答案草稿。
- 正确答案草稿。
- 干净题面图片。
- 识别状态。
- 去痕状态。
- 置信度或需复核标记。
- 模型调用记录，用于排查和重新处理。

原则：

- 原图优先保留。
- 题目区域选择优先于整图识别，避免多题照片误识别。
- 干净题面用于复习。
- 低置信度结果必须可编辑。
- 去痕策略必须保守，避免误删题目本身。

### 13.1 国内模型策略

真实模型接入的默认原则：

- 国内模型优先，优先考虑访问稳定性、中文/数学题识别效果、单位成本和合规可控性。
- 不把 GPT-5.5 或海外旗舰模型作为默认运行时识别投入。
- 业务层保持 provider-agnostic，只依赖统一的 `AI 能力契约`，不直接耦合某个供应商 SDK。
- 模型价格、名称和能力上线节奏会变化；正式接入前必须复核供应商官方文档和当期价格。

推荐第一候选链路：

1. `题目区域检测`：先用本地/前端 mock 完成 UI；真实接入时可由国内视觉模型或版面分析能力给出候选区域。
2. `题目 OCR`：优先评估阿里云百炼 Qwen 体系的 `qwen-vl-ocr-latest` 或同系列最新 OCR/视觉模型，用于题目区域文字、手写、公式、表格和图形相关内容提取。
3. `结构化整理`：优先评估 Qwen 轻量/Flash 系列，把 OCR 结果整理为题目 JSON、待复核项和干净题面渲染数据。
4. `疑难升级`：只有在复杂几何图、数学公式、低置信度、文本冲突或用户要求重新处理时，才升级到更强的国内模型，例如 Qwen Plus/Max 级别或同等能力模型。
5. `备选供应商`：火山引擎豆包视觉理解作为备选或后续 A/B 方案；不在第一版同时引入两套生产链路。

实现策略：

- 先接通一条国内模型链路，再考虑多供应商 A/B。
- 常规识别、疑难升级和失败降级必须有明确状态，让用户知道 AI 结果仍需复核。
- 干净题面优先由结构化数据和保留的图形片段重新渲染；避免把不可编辑的模型生成图当成唯一结果。

参考来源：

- 阿里云百炼视觉理解文档：<https://help.aliyun.com/zh/model-studio/vision-model>
- 阿里云百炼模型价格文档：<https://help.aliyun.com/zh/model-studio/model-pricing>
- 火山引擎豆包视觉理解最佳实践：<https://www.volcengine.com/docs/508/1398959>

## 14. 隐私与安全

上传图片可能包含姓名、学校、班级、字迹、分数、老师批注等敏感信息。MVP 应采用最小收集原则。

第一版约束：

- 不要求账号系统。
- 不要求云同步。
- 不默认把真实儿童学习照片发送到外部服务。
- 如后续接入外部 AI 服务，必须先明确上传提示、授权方式、存储位置、保留周期和删除机制。

## 15. UI 文案与语气

语言：

- 中文优先。
- 面向学生和家长都能理解。

语气：

- 清楚。
- 温和。
- 不恐吓用户。
- 不把 AI 结果包装成绝对正确。

避免：

- 在可见 UI 中大量使用 OCR、多模态、置信度等技术词。
- 让孩子直接面对复杂参数。
- 让未来应用占位喧宾夺主。

## 16. 成功指标

产品层面：

- 用户可以从 App Hub 进入错题收集应用。
- 用户可以完成一次上传、选择题目区域、检查、保存、进入错题本列表、查看详情的闭环。
- 保存记录默认可用于复习，而不是只保存一张带答案痕迹的照片。

体验层面：

- 用户能明确区分原图和干净题面。
- 用户能明确知道本次识别的是哪一道题，而不是整张多题照片。
- 用户能理解识别结果需要复核。
- 用户在识别失败时仍能手动保存。

后续规划层面：

- 文档和 UI 不把 EvoCraft 锁死为单一错题工具。
- 后续应用能自然接入 App Hub。

## 17. 验收标准

- App Hub 展示 EvoCraft 名称和 AI 学习助手应用集合定位。
- App Hub 中 `错题收集` 是当前可用应用。
- 用户可以上传一张图片并看到预览。
- 上传后用户可以进入独立 `选择题目区域` 步骤。
- 系统可以展示 AI 候选题目区域；如果没有候选结果，用户可以手动画框。
- 用户可以选择一个候选区域，或通过手动画框确认本次要识别的一道题。
- 用户可以对选中区域做基础拖动和缩放调整。
- 用户可以删除误识别或干扰手动画框的候选区域。
- 删除全部候选区域后，确认识别应保持不可用，直到用户手动画框或重新自动找题。
- 用户确认区域后，识别检查页只处理该区域。
- 用户可以选择语文、数学或英语。
- 系统可以展示识别草稿，或展示清晰的识别失败状态。
- 系统可以展示干净题面，或展示需复核/失败状态。
- 用户可以在保存前编辑识别字段。
- 用户可以保存一个错题记录。
- 用户可以从明确的 `错题本` 入口查看全部已保存错题。
- `错题本` 中至少能看到记录标题、科目、保存时间和去痕/复核状态。
- 用户可以从 `错题本` 打开某条已保存错题详情页。
- 用户可以打开已保存错题详情页。
- 详情页可以返回 `错题本`。
- 详情页默认展示干净题面。
- 详情页提供查看确认后题目区域和整张原图入口。
- 原图和确认后的题目区域都保留在记录中。
- 模型接入设计以国内模型优先、分层调用、provider-agnostic 为默认策略。
- 完成 MVP 不依赖账号、云同步、游戏化、多题拆分或其他学习应用。

## 18. 后续决策点

- 第一版 App Hub 是否在 UI 中展示未来应用占位，还是只展示当前可用应用。
- 第一版技术栈选择。
- 第一版存储采用浏览器本地存储、桌面本地文件，还是轻量数据库。
- 识别能力先继续 mock，还是接入阿里云百炼 Qwen 体系完成第一条国内 AI/OCR 链路。
- 火山引擎豆包视觉理解何时作为备选或 A/B 方案接入。
- 书写痕迹去除采用自动处理、人工框选辅助，还是先用 mock 结果完成 UI 闭环。
- 第一批目标年级是否聚焦小学高年级到初中。

## 19. UI 生成输入摘要

生成 UI 图时，应优先体现：

- EvoCraft 是应用集合，不是单一错题工具。
- 第一版只有 `错题收集` 可用。
- 上传后有独立 `选择题目区域` 步骤。
- 桌面端选题区域画布、候选框、手动画框、缩放和确认动作。
- 候选框删除动作，以及删光候选框后的手动画框/重新自动找题恢复路径。
- 识别复核页展示确认后的题目区域、干净题面和整张原图溯源入口。
- 明确的 `错题本` / 全部已保存错题列表入口。
- 原图和干净题面的区别。
- AI 结果是草稿，需要用户复核。
- 国内模型优先和分层调用是实现策略，不应在儿童可见 UI 中暴露复杂模型名称。
- 保存后的错题记录清晰、安静、适合复习。
