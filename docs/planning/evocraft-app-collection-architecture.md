# EvoCraft 应用集合架构

最后更新：2026-05-10

## 架构定位

EvoCraft 不是单一错题工具，而是面向孩子的 AI 学习助手应用集合。第一版 MVP 只实现其中第一个应用：`错题收集`。

顶层产品应保留应用集合结构，方便后续加入背单词、复习计划、阅读辅助、数学专项练习、虚拟人物与奖励系统等应用。

## 第一版边界

第一版需要保留大的架构，但不能让架构拖慢 MVP：

- 可以有 EvoCraft 顶层壳、应用入口、基础导航。
- 第一个可用应用是 `错题收集`。
- 其他应用可以作为未来方向记录在文档或低干扰占位入口中。
- 不实现其他应用的业务逻辑、数据流、复杂 UI。

## 推荐产品结构

```text
EvoCraft
├── App Hub / 应用集合首页
│   ├── 错题收集（MVP 可用）
│   ├── 背单词（未来）
│   ├── 复习计划（未来）
│   ├── 学习角色 / 装扮（未来）
│   └── 更多学习应用（未来）
├── Shared Learning Profile / 学习档案（未来）
├── Shared Reward Layer / 奖励层（未来）
└── Shared AI Services / AI 能力层（逐步接入）
```

## 设计原则

- `错题收集` 是第一个应用，不是 EvoCraft 的全部。
- 顶层导航要能自然承载多个学习应用。
- 每个学习应用应有独立入口、独立核心流程、清晰数据边界。
- 共享能力可以逐步沉淀，例如图片识别、学习记录、奖励、用户档案、AI 分析。
- 第一版不要为了未来应用提前实现复杂插件系统；先用清晰的信息架构和命名保留扩展空间。

## 对 MVP UI 的影响

第一版 UI 应包含一个轻量的 EvoCraft 应用集合入口：

- 显示 EvoCraft 名称和“AI 学习助手应用集合”的定位。
- 明确 `错题收集` 是当前可用的第一个应用。
- 未来应用可以用低优先级方式展示，例如“背单词 - 规划中”，但不能干扰主流程。
- 进入 `错题收集` 后，用户应专注完成上传、识别、去痕、编辑、保存这一条闭环。

## 后续扩展记录

新增任何应用前，应先补充对应 PRD，至少明确：

- 应用名称。
- 目标用户场景。
- 输入和输出。
- 与学习档案、错题、奖励层的关系。
- 是否复用已有 AI 能力。
- MVP 边界和非目标。
